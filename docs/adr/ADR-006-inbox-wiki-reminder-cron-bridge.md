# ADR-006: Inbox-WIKI Phase-D.a — reminder_job: Stage-0 fast-path (not the Stage-1a router), jobs.Job + APScheduler dual-store, module-level firing context

**Status:** accepted
**Date:** 2026-05-12
**Context:** [D-002](../Spec-WIKI/questions/Q-B-02-jobs-db.md), [D-010](../Spec-WIKI/questions/Q-B-10-nl-time-parsing.md), [D-022](../Spec-WIKI/questions/Q-B-14-voice-photo-input.md), [D-023](../Spec-WIKI/questions/Q-B-12-tg-confirmations.md), [smart-inbox-routing](../Spec-WIKI/concepts/smart-inbox-routing.md), [ADR-005](ADR-005-inbox-wiki-route-confirm.md), tech-spec §3/§6; bd `aisw-kcz` (Inbox-WIKI Phase-D.a), epic `aisw-t2r`, blocks `aisw-19o` (Phase-D.b digest_job). Discovery/design/plan: `docs/superpowers/specs|plans/20260512-inbox-wiki-cron-bridge-{discovery,design,plan}.md` (T-1..T-8, D-OQ-1/2/3/4, D-R-1, R-2).

## Context

The MVP scheduler core (Phase-4) ships `build_scheduler` (APScheduler `AsyncIOScheduler` + `SQLAlchemyJobStore` on `jobs.db`), a 5-lane `PriorityJobQueue`, a `jobs.jobs` ORM table — but nothing creates a one-shot user-facing job. ADR-005's consequence 3 named this: *"Phase-D (`aisw-kcz`) hooks in here: a RouterDecision carrying a reminder/aggregator spec will … also drive a jobs.db insert from the same `_handle_route_confirm` path."* Phase-D.a delivers the one-shot reminder slice («разбуди завтра в 6 позвонить врачу»); recurring digests («каждый день в 9 сводка») split out to `aisw-19o` (Phase-D.b).

## Options & decisions

1. **Where does a reminder get detected — the Stage-0 Haiku classifier, or the Stage-1a Inbox-WIKI router?** The bd issue title says "reminder_job" without saying which. tech-spec §6 describes a Stage-0 *fast-path*: a high-confidence `intent=reminder` short-circuits before the router/librarian (no Claude run needed to schedule a reminder). → **Stage-0 fast-path.** `DefaultPipeline._run_text_pipeline` branches on `ClassifierResult.intent is Intent.REMINDER and confidence ≥ 0.85` **before** `_ROUTABLE_INTENTS`/the Stage-1a router; `prompts/inbox.md` and the router are untouched. Rejected: routing reminders through the librarian — wasteful (a Claude run to parse "в 6"), and the librarian is for WIKI mutations, not job scheduling. *(D-OQ-1.)* This deliberately contradicts a literal reading of the bd title's "RouterDecision-carries-reminder" phrasing; recorded here because it is a non-obvious call.
2. **Confirm or implicit-ack?** → **Explicit inline-button confirm**, reusing the Phase-C machinery (`ConfirmationService.request_explicit(draft, keyboard_factory=build_route_confirm_keyboard)` with `category='reminder'`). The `jobs.Job` row is created **only** on the confirm callback (`on_confirm_callback` → `_handle_reminder_confirm`). Rejected: implicit-ack (schedule immediately, let the user cancel) — D-023 reserves implicit-ack for low-risk reads, and a misparsed time firing at 3 AM is exactly what a confirm prevents. *(D-OQ-2.)* No new keyboard — the 2-button «✅ Подтвердить / ❌ Отмена» from Phase-C is reused.
3. **SSoT for a scheduled reminder: `jobs.Job` row or the APScheduler trigger?** → **`jobs.jobs` is the domain SSoT** (`kind='reminder_job'`, `status`, `scheduled_at_utc`, `payload=ReminderPayload(...).model_dump(mode='json')`, `priority=Lane.USER_WRITE`); APScheduler's `SQLAlchemyJobStore` (same `jobs.db`) holds the trigger. `create_reminder_job` writes + **commits** the `Job` row, **then** `scheduler.add_job(...)` — so a crash in the millisecond gap leaves at worst a `pending` row with no trigger (the reminder silently does not fire), never a trigger pointing at a missing row. **No reconciliation pass in the MVP** (documented limitation; a startup `jobs.jobs ↔ APScheduler` reconciler is a later ops task). *(D-OQ-4 / R-1.)* No new table, no Alembic migration — `jobs.jobs` columns suffice.
4. **How does the fire-time callback reach the aiogram bot + a DB session?** `SQLAlchemyJobStore` pickles each job's callable reference + args, so the callable must be a stable module-level function taking only picklable args. → **`ai_steward_wiki.scheduler.firing:fire_job(job_id: int)`** — one picklable int; the `(TgSender, jobs async_sessionmaker)` pair is read from a module-level registry installed once at startup by `set_firing_context(...)` (called in `__main__` right after the bot is built). `fire_job` raises `FiringNotInitialisedError` (fail-fast) if the registry was never set. *(D-OQ-3.)* This deliberately differs from `scheduler/maintenance.py`'s retention jobs, which pass non-picklable `session_maker`s as `add_job` args and get away with it because they are re-registered on every boot with `replace_existing=True`; a one-shot reminder cannot be re-registered, so it must persist with picklable args only.
5. **Bare past wall-clock time vs explicitly-past absolute date.** `classifier.time_parse.parse_time` gains `prefer_future: bool = False` → `dateparser` `PREFER_DATES_FROM='future'`; the reminder path calls it with `prefer_future=True`, so «разбуди в 6» at 21:00 rolls forward to tomorrow 06:00 (TZ-aware → DST-safe). User TZ from `users.toml` `UserRecord.tz`, falling back to a new `Settings.default_user_tz = "Europe/Moscow"`. After `prefer_future=True`, if `when_utc` is still `≤ now` it was an explicit past *date* («напомни 5 мая» on 12 мая) → ru rejection («Эта дата уже прошла — назови будущую.»), no job. Roll-forward of explicitly-past absolute dates is out of scope. *(D-R-1.)*
6. **Reminder text vs raw text.** `ReminderPayload.message = distilled_payload['reminder_text']` if it is a non-empty `str`, else the raw user text. `prompts/classifier.md` is **not** bumped — its `distilled_payload` is already an opaque bag of extracted entities, so reading an optional `reminder_text` opportunistically needs no prompt change. *(D-OQ-1 / R-2.)*
7. **Misfire handling.** `misfire_grace_time=None` (= run as soon as possible after a missed fire) so a reminder missed during downtime still arrives (late); `coalesce=True` (job_defaults from `build_scheduler`) collapses a backlog to one fire; an `EVENT_JOB_MISSED` listener in `__main__` logs `scheduler.reminder.misfired`. No DLQ/retry for `reminder_job` (single-shot, TG-deliver-only; strict DLQ parity with tech-spec §3 is deferred).
8. **Recurring-digest phrasing.** Detected by a small ru keyword set («кажд», «ежедневн», «еженедельн», «сводк», «дайджест») → ru "not yet" reply, no job. Precise recurrence recognition + `CronTrigger` + `--add-dir` multi-WIKI is `aisw-19o`'s job (`DigestPayload` already exists).

## Consequences

1. **New module `M-SCHEDULER-FIRING`** (`scheduler/firing.py`: `set_firing_context`, `create_reminder_job`, `fire_job`, `FiringNotInitialisedError`) — depends on `M-STORAGE-JOBS`, `M-SCHEDULER` (`Lane`), `M-TG-TEXT` (`TgSender`, typing-only). `M-STORAGE-JOBS` MODULE_MAP gains `ReminderPayload`; `M-CLASSIFIER-STAGE0` `parse_time` gains `prefer_future`; `M-FOUNDATION-SETTINGS` gains `default_user_tz`; `M-TG-PIPELINE-CLASSIFIER` gains the reminder fast-path + `_handle_reminder_confirm` + `build_reminder_recap` + the `REMINDER_*` ru constants + a `TimeParser` Protocol + new optional `DefaultPipeline` deps (`time_parser`, `jobs_session_maker`, `scheduler`, `user_tz_lookup`, `default_user_tz`, `clock`); `M-RUNTIME-WIRING` (`__main__`) gains `_TimeParserAdapter`, `_on_job_missed`, the `firing.set_firing_context` call, the `EVENT_JOB_MISSED` listener, and a `users.toml`-backed `_user_tz_lookup`. New log anchors `tg.pipeline.reminder.{detected,unparseable,rejected_past,recurring_not_yet,confirm_requested,confirm_created,confirm_cancelled,confirm_stale}`, `tg.pipeline.confirm.reminder_dispatched`, `scheduler.reminder.{scheduled,fired,delivered,deliver_failed,skipped,misfired}`.
2. **No new third-party dependency, no new SQLite table, no new Alembic migration, no `prompts/*` change.** One new `Settings` field (`default_user_tz`).
3. **Known limitations (accepted, MVP):** the millisecond-gap silent-miss (no `jobs.jobs ↔ APScheduler` reconciler); no DLQ/retry for `reminder_job`; no roll-forward of explicitly-past absolute dates (rejected instead); reminder management UX (`/reminders` list/cancel/snooze/edit, post-create cancel button) is unbuilt; recurring digests are stubbed out to `aisw-19o`.
4. **`aisw-19o` (Phase-D.b) hooks in here:** `_handle_reminder_intent`'s recurring-keyword branch is the placeholder; the digest phase will replace it with proper recurrence parsing → `CronTrigger` → `DigestPayload` → a Claude run with `--add-dir` into the relevant Domain-WIKIs, reusing this module's `set_firing_context`/firing pattern.
