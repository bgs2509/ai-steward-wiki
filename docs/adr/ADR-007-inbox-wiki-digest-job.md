# ADR-007: Inbox-WIKI Phase-D.b.1 — digest_job: direct-fire under run_wiki_session's lock (no PriorityQueue consumer), Recurrence value, dedicated context registry

**Status:** accepted
**Date:** 2026-05-12
**Context:** [D-002](../Spec-WIKI/research/tech-spec-draft.md), [D-019](../Spec-WIKI/decisions/D-019-cron-failure-mode.md), [D-024](../Spec-WIKI/decisions/D-024-digest-format.md), [D-025](../Spec-WIKI/decisions/D-025-output-size.md), [smart-inbox-routing](../Spec-WIKI/concepts/smart-inbox-routing.md), tech-spec §3; [ADR-006](ADR-006-inbox-wiki-reminder-cron-bridge.md); bd `aisw-oqq` (Phase-D.b.1), parent `aisw-19o` (Phase-D.b umbrella), epic `aisw-t2r`, blocks `aisw-w3k` (Phase-D.b.2 D-024 presentation). Discovery/design/plan: `docs/superpowers/specs|plans/20260512-inbox-wiki-digest-job-{discovery,design,plan}.md`.

## Context

ADR-006 shipped the one-shot `reminder_job` (no Claude) and stubbed recurring digests («каждый день в 9 сводка») out to `aisw-19o`. `aisw-19o` (Phase-D.b) bundled three not-yet-built sub-systems (NL recurrence parsing; the asyncio.PriorityQueue worker-loop consumer; WIKI-set `--add-dir` resolution) plus `DigestPayload` widening, D-024 presentation, and TG delivery — too large for one context window per the Plan-Sizing budget. It is split into **aisw-oqq** (this — the runnable vertical slice) and **aisw-w3k** (D-024 presentation polish).

## Options & decisions

1. **digest_job execution path: the spec's PriorityQueue producer/consumer, or direct-fire?** tech-spec §3 mandates APScheduler → `asyncio.PriorityQueue` → a worker pool under `Semaphore(MAX_CONCURRENT_CLI=4)` + per-WIKI lock. → **Direct-fire.** `fire_digest_job` (the picklable APScheduler callback) calls the digest runner adapter directly — exactly how `scheduler/maintenance.py` runs its jobs. Rejected: building the `PriorityJobQueue` consumer now — it has **no consumer today** (`/cron_add` wiki_job not built; the interactive path needs streaming and is not a fit for a deferred queue), so it would be infrastructure with zero user value (YAGNI). **The PriorityQueue worker-loop is de-scoped out of `aisw-19o` entirely** → its own future bd issue. Concurrency control is not lost: the digest runner adapter calls `wiki.runner.run_wiki_session`, which already acquires the per-WIKI lock (semaphore → memlock → flock) via its `LockAcquirer` — so `fire_digest_job` does **not** take a lock itself (avoids double-locking).
2. **Recurrence representation.** → A small frozen Pydantic value `Recurrence(kind: Literal["daily","weekly"], time_hhmm, weekdays: tuple[int,...], tz)` in a new module `classifier/recurrence.py`, with `to_cron()` → APScheduler `CronTrigger` kwargs (`daily` → `{hour, minute}`; `weekly` → `{day_of_week: "mon,fri", hour, minute}`), serialised into `DigestPayload`. Reusable by future `cron_user`/`tracker_*` kinds. `parse_recurrence(text, *, user_tz, correlation_id="")` is a **conservative rule-based ru regex** (daily / weekly-by-weekdays incl. «по будням»/«по выходным»/named days) returning `RecurrenceParseResult(recurrence | None, escalate, reason)`; monthly / interval / raw-cron / ambiguous → `escalate`. Rejected: extending `classifier/time_parse.py` (single-instant, different shape); a Haiku-backed parser as the primary path (the rule-based one covers the MVP phrasings; `prompts/recurrence.md` is shipped for a later wiring but not used in this phase). *(Open executor decisions OQ-1/OQ-2 resolved this way.)*
3. **`DigestPayload` widening.** `kind='digest'` was `{wiki_id: str, window_hours: int}` with **no producer anywhere** (verified) → freely replaced: `wiki_scope: Literal["all"] = "all"`, `recurrence: Recurrence`, `window_hours: int = Field(24, ge=1, le=168)`, `prompt_hint: str | None`; `extra='forbid'` + `frozen=True` kept. No Alembic migration (JSON payload widening; the recurrence trigger is owned by `SQLAlchemyJobStore` in `jobs.db`). The `jobs.jobs` schema is unchanged — it has no `enabled`/`cron_expr` columns; the `Job` row is the domain SSoT (`status` ∈ `scheduled` → `disabled`, `retry_count` for strikes), the APScheduler jobstore holds the trigger; commit-before-add_job ordering (same invariant as ADR-006). *(OQ resolved: no schema migration.)*
4. **Fire-time context.** Mirrors ADR-006's `fire_job` registry, but a **separate** `set_digest_context(*, scheduler, runner, resolve_owner_wikis, jobs_session_maker, sender)` registry (the digest callback needs more deps than the reminder one — a runner adapter, an owner-WIKI-set resolver, and the scheduler handle so it can `remove_job` on auto-disable). `fire_digest_job(job_id: int)` takes only a picklable int; `DigestNotInitialisedError` (fail-fast) if the registry was never installed. *(OQ resolved: `set_digest_context` carries `scheduler`; deps reached via the registry, not closed over.)*
5. **WIKI-set resolution.** MVP: an `'all'` sentinel — `_resolve_owner_wikis_factory(wiki_root)` globs `<wiki_root>/<owner_telegram_id>/*-WIKI/`, excludes `Inbox-WIKI`, returns `[(dir_name, path), …]`. The first is the **primary** `wiki_path` (where Claude runs + the per-WIKI lock is held); the rest are read-only `--add-dir` context. `wiki.runner.run_wiki_session` gains `extra_add_dirs: list[Path] | None` appended to the CLI argv `--add-dir` list after the primary, alongside the existing `media_dirs`. Empty WIKI set → a friendly ru line («У тебя пока нет ни одной WIKI для сводки.»), the row is marked finished, **no strike, no disable** (not a failure). Named-subset selection in the same turn → `aisw-w3k`. *(OQ resolved: lock the primary WIKI only via run_wiki_session's acquirer.)*
6. **Delivery in this phase.** `fire_digest_job` delivers the Stage-1 assistant text via a **plain `sender.send_message`** (with a ~4000-char safety truncation); `prompts/digest.md` asks Claude for a scan-friendly ru summary (TL;DR first, sections, empty-digest line «🌿 Сегодня дел нет.»). The D-024/D-025 presentation contract — TL;DR-as-a-section, actionable inline cards for ±2h items, `<b>`-header section-boundary 4096 split with `(n/m)` markers, `send_document` fallback, the audit `run_outputs` row via `tg/output.deliver_output`, `/expand`, `/digest_now`, per-user toggles — is **all `aisw-w3k`**. *(OQ-3: the design recommended `deliver_output`; deferred to `aisw-w3k` to keep this phase's `set_digest_context` surface small — flagged as a small follow-up.)*
7. **Failure handling.** `fire_digest_job` catches every exception (the scheduler/event loop must stay alive): a run/delivery failure or timeout bumps `Job.retry_count`; the **3rd** consecutive failure sets `status='disabled'`, `scheduler.remove_job(f'digest:{id}')` (suppressing `JobLookupError`), and `move_to_dlq` (D-019; a `WikiRunnerTimeoutError` from `run_wiki_session`'s 600s kill-sequence counts as a strike). A bad/parse-failing payload → `disabled` + DLQ immediately. A successful run keeps `status='scheduled'`, resets `retry_count=0`. No retry-with-backoff in the MVP — 3 strikes → off.
8. **TG entry point.** The recurring-digest keyword branch in `tg/pipeline.py:_handle_reminder_intent` (which ADR-006 left as a «пока не умею» stub) becomes a real `_handle_digest_intent`: it requires a wired `recurrence_parser` (else it falls back to the legacy "not yet" line); parses → on `escalate` a ru clarification, else a `category='digest'` `PendingConfirmDraft` (`recurrence` serialised, `wiki_scope='all'`, `window_hours=24`) + recap → `ConfirmationService.request_explicit` with the existing 2-button keyboard; `on_confirm_callback` dispatches `category=='digest'` → `_handle_digest_confirm` → on `confirmed` `scheduler.firing.create_digest_job` + ru ack. The fast-path is still gated on `intent is Intent.REMINDER and confidence ≥ 0.85 and time_parser is not None` (ADR-006's gate) — so in production `time_parser` and `recurrence_parser` are both wired; a `recurrence_parser`-only wiring would be dead (accepted coupling for the MVP).

## Consequences

1. **New module `M-CLASSIFIER-RECURRENCE`** (`classifier/recurrence.py`: `Recurrence`, `RecurrenceParseResult`, `parse_recurrence`). **`M-SCHEDULER-FIRING`** gains `set_digest_context`, `create_digest_job`, `fire_digest_job`, `DigestRunner` Protocol, `DigestNotInitialisedError` (depends `+M-WIKI-RUNNER`, `+M-WIKI-LIFECYCLE`, `+M-CLASSIFIER-RECURRENCE`, `+ai_steward_wiki.scheduler.dlq.move_to_dlq`). **`M-STORAGE-JOBS`** `DigestPayload` widened (imports `Recurrence`). **`M-WIKI-RUNNER`** `run_wiki_session`/`_build_argv` gain `extra_add_dirs`. **`M-TG-PIPELINE-CLASSIFIER`** gains `_handle_digest_intent`, `_handle_digest_confirm`, `build_digest_recap`, `humanize_recurrence`, `RecurrenceParser` Protocol, the `DIGEST_*` ru constants, a `recurrence_parser` `DefaultPipeline` kwarg, the `category=='digest'` `on_confirm_callback` dispatch. **`M-RUNTIME-WIRING`** (`__main__`) gains `_RecurrenceParserAdapter`, `_DigestRunnerAdapter`, `_resolve_owner_wikis_factory`, the `firing.set_digest_context(...)` call, and passes `recurrence_parser=`. New `prompts/digest.md` + `prompts/recurrence.md` (the latter shipped but not wired). New log anchors `classifier.recurrence.parse`, `tg.pipeline.digest.{detected,unparseable,confirm_requested,confirm_created,confirm_cancelled,confirm_stale}`, `tg.pipeline.confirm.digest_dispatched`, `scheduler.digest.{scheduled,fired,delivered,failed,disabled,skipped}`.
2. **No new third-party dependency, no new SQLite table, no new Alembic migration.** No new `Settings` field (`default_user_tz` from ADR-006 is reused for the recurrence TZ fallback via the pipeline's `_resolve_user_tz`).
3. **Known limitations (accepted, MVP):** the PriorityQueue worker-loop consumer is unbuilt (de-scoped out of `aisw-19o`); recurrence is daily/weekly only (monthly/interval/cron-string → escalate); the Haiku-fallback recurrence path is a stub (`prompts/recurrence.md` shipped, not wired); the digest is delivered via a plain truncated `send_message` (D-024/D-025 presentation + `tg/output.deliver_output` audit row → `aisw-w3k`); planner-semantics context is a one-line note, not a real `jobs.db` query (→ `aisw-w3k`); the millisecond-gap silent-miss (no `jobs.jobs ↔ APScheduler` reconciler) carries over from ADR-006; digest management UX (`/jobs_list`, cancel/snooze/edit) is unbuilt.
4. **`aisw-w3k` (Phase-D.b.2) hooks in here:** it replaces the plain `send_message` in `fire_digest_job` with the D-024 presentation contract (`deliver_output`, cards, section split) and the one-line planner context with a real `jobs.db` window query, and adds named-subset WIKI selection + `/expand` + `/digest_now` + per-user toggles.
