# ADR-024: Inbox-WIKI Phase-D.b.2a — digest presentation core: route fire_digest_job through tg.output.deliver_output(kind='digest'); real jobs.db planner-window context; D-024 HTML+TL;DR prompt contract

**Status:** accepted
**Date:** 2026-05-12
**Context:** [D-024](../Spec-WIKI/decisions/D-024-digest-format.md), [D-025](../Spec-WIKI/decisions/D-025-output-size.md), [smart-inbox-routing](../Spec-WIKI/concepts/smart-inbox-routing.md), tech-spec §3/§6; [ADR-006](ADR-006-inbox-wiki-reminder-cron-bridge.md), [ADR-007](ADR-007-inbox-wiki-digest-job.md); bd `aisw-w3k` (Phase-D.b.2a), sibling `aisw-269` (Phase-D.b.2b — cards / `/expand` / `/digest_now` / toggles / named-subset WIKI), parent `aisw-19o`, epic `aisw-t2r`. Discovery/design/plan: `docs/superpowers/specs|plans/20260512-inbox-wiki-digest-presentation-{discovery,design,plan}.md`.

## Context

`aisw-oqq` (Phase-D.b.1, ADR-007) shipped the runnable digest vertical slice but deliberately deferred the D-024/D-025 *presentation* contract: it delivered the Stage-1 text via a plain truncated `sender.send_message`, with a one-line `planner_context` stub. `aisw-19o` (Phase-D.b umbrella) bundled the whole presentation polish plus the interactive surface (cards, `/expand`, `/digest_now`, per-user toggles, named-subset WIKI) — too large for one context window per the Plan-Sizing budget. It is split: **aisw-w3k** (this — presentation core: delivery routing + planner query + prompt contract) and **aisw-269** (the interactive half).

## Options & decisions

1. **Digest delivery: a bespoke digest formatter/splitter, or reuse `tg.output.deliver_output`?** → **Reuse `deliver_output(kind='digest')`.** `tg/output.py` already implements everything the D-024 *delivery* side asks for: `ChainSplitter` lists `<b>` headers as the top boundary, emits `(i/M)` continuity footers, has the `>10000` → `LengthCapSummarizer`/Haiku-summary + `send_document` fallthrough, atomically persists the full text to `<wiki>/data/runs/<date>/<run_id>.md` with frontmatter, and writes the `audit.run_outputs` index row. Building a parallel digest splitter would duplicate `ChainSplitter` and drift (DRY). So `fire_digest_job` now generates `run_id = f"digest-{uuid4().hex[:12]}"`, sets `runs_dir = primary_path / "data" / "runs"`, and calls `deliver_output(... kind="digest", job_id=job_id)`; the `>4096` truncation (`_DIGEST_TG_LIMIT`) is removed — `deliver_output` owns sizing. The empty-WIKI branch (`_DIGEST_NO_WIKI_RU`) stays a plain `sender.send_message` — it is a control message, not a Claude output, so no `data/runs/` persist and no audit row.
2. **`set_digest_context` surface.** → It gains one required dep: `audit_session_maker: async_sessionmaker[AsyncSession]` (audit.db, for the `run_outputs` row); `_digest_ctx` becomes a 6-tuple `(scheduler, runner, resolve_owner_wikis, jobs_session_maker, audit_session_maker, sender)`. **No `summarizer` added** — `deliver_output`'s default `LengthCapSummarizer` is used (the same default `_OutputDeliveryAdapter` uses today); wiring a real Haiku post-step summarizer just for digests now would be premature (YAGNI) — it is a project-wide concern. `__main__` passes `audit_session_maker=audit_maker`.
3. **Planner context.** → A module-private `async _build_planner_context(session, *, owner_telegram_id, window_hours, now_utc, tz) -> str` in `scheduler/firing.py` (it already imports `Job`, `parse_job_payload`, and owns the jobs session — KISS, no new `scheduler/digest_planner.py` module). It selects `jobs.Job` rows with `status=='scheduled'` AND `scheduled_at_utc IS NOT NULL` AND `scheduled_at_utc <= now_utc + window_hours`, ordered by `scheduled_at_utc`, and renders «- HH:MM — <title>» lines in the owner's tz (`title` = the payload's `message`/`prompt_hint`, falling back to `job.kind`; `tz` = the digest job's own `payload.recurrence.tz`). Empty → «На ближайшие N ч ничего не запланировано.». Recurring jobs (`scheduled_at_utc IS NULL`, cron-driven) are excluded — surfacing "your digest itself fires daily at 09:00" is noise. A new DEBUG log anchor `scheduler.digest.planner_context` (with `n_planned`).
4. **`prompts/digest.md` D-024 contract.** → Rewritten (`semver: 0.0.1` → `0.1.0`): `<b>📌 TL;DR</b>` as the **first** section (3–5 lines), then only-if-content `<b>📅 Сегодня</b>` / `<b>💊 Лекарства</b>` / `<b>📈 Трекеры</b>` / `<b>📝 Обновления WIKI</b>`; HTML whitelist `<b><i><u><s><code><pre><a><blockquote>`, escape `< > &` in literal text, **never MarkdownV2**; one short line per item, no tables; empty ⇒ exactly «🌿 Сегодня дел нет.»; consume the «Запланировано…» block from the user message; read-only, never edit files.

## Consequences

1. **`M-SCHEDULER-FIRING`** (`scheduler/firing.py`, `VERSION 0.2.0 → 0.3.0`): `fire_digest_job` delivers via `tg.output.deliver_output(kind='digest')`; `set_digest_context` gains `audit_session_maker`; new private `_build_planner_context`; `_DIGEST_TG_LIMIT` removed; DEPENDS `+ai_steward_wiki.tg.output.deliver_output`. New log anchor `scheduler.digest.planner_context`; `scheduler.digest.delivered` now carries `run_id`/`n_messages`/`document_sent`. **`M-RUNTIME-WIRING`** (`__main__.py`, `v0.5.0 → v0.5.1`): `firing.set_digest_context(..., audit_session_maker=audit_maker)`; `prompts/digest.md` rewritten. **`M-TG-TEXT`** (`tg/output.py`): unchanged — only newly *used* by `fire_digest_job` and newly *tested* with digest-flavoured inputs.
2. **No new third-party dependency, no new SQLite table, no new Alembic migration, no new module.** FR-2 (section split + `(n/m)` + `send_document` fallthrough) needed **no production code** — `ChainSplitter`/`deliver_output` already do it; covered by two new `tests/unit/tg/test_output.py` cases.
3. **Out of scope (→ `aisw-269`, Phase-D.b.2b):** actionable inline cards for ±2h items (medication/event/pending_confirmation) + callback handlers; `/expand <section>`; `/digest_now`; per-user section toggles (+ first `alembic/sessions/0002_*`); named-subset WIKI selection at digest-creation time. Out of scope project-wide: a real Haiku post-step summarizer wiring; the `jobs.jobs ↔ APScheduler` reconciler for the millisecond-gap silent-miss (carried from ADR-006); `data/runs/` retention.
4. **Rejected alternatives:** a bespoke digest splitter/formatter module (DRY — `ChainSplitter` already prioritises `<b>`); adding a `HaikuSummarizer` to the digest registry now (YAGNI — project-wide, not digest-specific); a `scheduler/digest_planner.py` module for the jobs query (KISS — `firing.py` already has everything).
