# ADR-005: Inbox-WIKI Phase-C — confirm-before-route replaces Phase-B auto-execute; reuse `pending_confirms`; no `proposed_actions` field; no toggle

**Status:** accepted
**Date:** 2026-05-12
**Context:** [D-023](../Spec-WIKI/questions/Q-B-12-tg-confirmations.md), [D-022](../Spec-WIKI/questions/Q-B-14-voice-photo-input.md), [smart-inbox-routing](../Spec-WIKI/concepts/smart-inbox-routing.md), [ADR-004](ADR-004-inbox-wiki-route-ingest.md); bd `aisw-e45` (Inbox-WIKI Phase-C), epic `aisw-t2r`. Discovery/design/plan: `docs/superpowers/specs|plans/20260512-inbox-wiki-route-confirm-*.md`.

## Context

Phase-B (ADR-004) wired the routable branch in `tg/pipeline.py`: a `ROUTE`/`CREATE_WIKI` `RouterDecision` is **executed immediately** — resolve/create the target `<Domain>-WIKI`, move the raw payload in, run the Stage-1b "librarian" Claude there, reply with `notes + summary`. ADR-004 §3 / consequence 3 explicitly named the missing gate: *"Phase-B auto-executes route+ingest with no user confirmation — the inline-button confirm loop is Phase-C."* Phase-C closes that gap at the seam ADR-004 deliberately left between `router.route()` and `librarian.ingest()`.

## Options & decisions

1. **Where to persist the staged action.** The graduated-confirmation machinery (D-023, `tg/confirm.py`) already persists explicit-level drafts in `sessions.pending_confirms` with a 10-minute TTL, a `category` and a `draft_json` blob (added in chunk 10). → **Reuse `pending_confirms`** with `category='route_ingest'` and `draft_json = json.dumps(route_action_payload)`. Rejected: a new `pending_router_action` table — it would duplicate the TTL/status machine and the sweeper for no benefit. **No Alembic migration.**
2. **Structured `proposed_actions` on `RouterDecision`?** → **No.** The single proposed action for Phase-C is fully derivable from `(intent, target_wiki)` (`build_route_recap` formats the ru recap). A structured list of actions (route + reminder + aggregator) only becomes necessary with the Phase-D cron bridge (`aisw-kcz`); adding it now is YAGNI and would force a `RouterDecision` schema + `prompts/inbox.md` change with no current consumer. The bd issue title's "`RouterDecision.proposed_actions`" is read as *"the action proposed by the RouterDecision"*, not a literal new field. The draft schema (`route_action_to_payload`) is, however, forward-compatible — Phase-D can extend the payload dict.
3. **Replace Phase-B's auto-execute, or gate it behind a flag?** → **Replace.** Confirm-before-mutate is strictly the desired behaviour (ADR-004 named it as the missing piece). A `route_confirm_enabled` toggle would be dead config with two code paths to test. KISS.
4. **Keyboard.** → **A new 2-button keyboard** (`build_route_confirm_keyboard`: «✅ Подтвердить» / «❌ Отмена`) for route confirms; the chunk-10 explicit-confirm path keeps its 3-button keyboard (with «✏️ Изменить»). The «Изменить»/re-route UX isn't meaningful for routing in the MVP and is omitted. `ConfirmationService.request_explicit` grows one additive param — `keyboard_factory: Callable[[int], object] = build_explicit_keyboard` — instead of a parallel method. `tg/handlers.py` is untouched (`parse_confirm_callback` already accepts `confirm`/`cancel`; the route-vs-legacy dispatch happens inside `DefaultPipeline.on_confirm_callback` by reading the pending row's `category`).
5. **Slow Stage-1b on the callback.** The move+ingest now runs in `on_confirm_callback`, not on the original message (up to ~300s). → The aiogram callback is already `answer()`-ed in `handlers.py`; the pipeline additionally sends a short ru ack («📝 Записываю в вики…») **before** the ingest so the user isn't left hanging. The recap message's keyboard is **not** removed afterwards (no `edit_reply_markup` on `TgSender`; a repeat tap is harmless — `resolve` is race-safe `UPDATE … WHERE status='pending'` → returns `None` → "stale" reply).
6. **Cancel / TTL expiry.** → **The staged raw stays in `Inbox-WIKI/raw/`** (D-022 / ADR-004's forward-only, no-rollback philosophy — it's a small audit artifact, content-addressed for media; a sweep job is a later ops task). `expire_due` (the existing sweeper) flips the row to `expired`. No user notification on TTL expiry (acceptable MVP).
7. **Where the (de)serialiser lives.** → **`inbox/route.py`** (`RouteAction` frozen dataclass + `route_action_to_payload` / `route_action_from_payload`) — it already owns `RouterDecision`; the helpers are pure and unit-testable in isolation. The ru recap/ack strings + `build_route_recap` live in `tg/pipeline.py` next to the other `*_RU` constants.

## Consequences

1. `M-INBOX-ROUTE` gains `RouteAction` + `route_action_to/from_payload`; `M-TG-TEXT` gains `build_route_confirm_keyboard` + the `request_explicit` `keyboard_factory` param; `M-TG-PIPELINE-CLASSIFIER` already depended on `M-INBOX-ROUTE` and the `M-TG-TEXT` CrossLink relation now also covers the confirm loop. New log anchors `tg.pipeline.route.confirm_requested|confirm_executed|confirm_cancelled|confirm_stale` and `tg.pipeline.confirm.route_dispatched`; Phase-B's `tg.pipeline.route.ingest_dispatched|delivered` are removed.
2. A routable turn now needs **two user interactions** — message → recap → tap → ingest. The Stage-1b run (and its per-WIKI lock) happens on the callback, not the original message. Accepted: the WIKI mutation is now gated by an explicit human confirm, which is the point.
3. **Phase-D (`aisw-kcz`) hooks in here:** the confirm callback currently replays only the Phase-B move+ingest; a `RouterDecision` carrying a reminder/aggregator spec will, in Phase-D, also (or instead) drive a `jobs.db` insert from the same `_handle_route_confirm` path, using an extended `route_action` payload.
4. Open follow-ups (not Phase-C): a structured `proposed_actions` list (Phase-D); the «Изменить»/re-route UX; TTL-expiry user notification; restore-from-trash on a `ROUTE` to a soft-deleted name; removing the recap keyboard on resolution (needs an `edit_reply_markup` on `TgSender`).
