# ADR-004: Inbox-WIKI Phase-B route+ingest — auto-create on missing ROUTE target, move-raw-before-Stage-1b, dedicated Librarian adapter

**Status:** accepted
**Date:** 2026-05-12
**Context:** [D-004](../Spec-WIKI/decisions/D-004-inbox-wiki-scope.md), [smart-inbox-routing](../Spec-WIKI/concepts/smart-inbox-routing.md), [ADR-003](ADR-003-inbox-wiki-router-invocation.md); bd `aisw-zd9` (Inbox-WIKI Phase-B), epic `aisw-t2r`. Discovery/design/plan: `docs/superpowers/specs|plans/20260512-inbox-wiki-route-ingest-*.md`.

## Context

Phase-A (ADR-003) ran the Stage-1a Router in `Inbox-WIKI/` and produced a `RouterDecision`, but did nothing with it — the bot just echoed `decision.notes`. Phase-B closes the loop: a `ROUTE`/`CREATE_WIKI` decision must select/create the target `<Domain>-WIKI`, move the raw payload into it, run a Stage-1b "librarian" session there, and reply with the ingest result. Six design forks were resolved via `/questions-answers` (2026-05-12).

## Options & decisions

1. **`ROUTE` to a target that doesn't exist.** The Router can hallucinate a name. → **Auto-create it** (`WikiLifecycleManager.create_wiki(owner, target, "_default")`) + log `inbox.route.route_target_was_missing` (warning), then proceed as `CREATE_WIKI`. Rationale: keeps the loop closed (losing user input is the worst outcome); `create_wiki`'s built-in Levenshtein near-dup absorbs typos; the reject/clarify alternative is heavier and overlaps Phase-C's confirm gate.
2. **When to move the raw payload into `<Domain>-WIKI/raw/`.** → **Move (write the text sidecar + `promote_path_to_raw` the media) BEFORE the Stage-1b run.** Stage-1b reads the raw from inside the target WIKI (`--add-dir`). On `WikiRunnerError` the moved raw stays there un-ingested (logged `inbox.route.ingest_failed`); a retry/sweep is a later ops task. Forward-only + idempotent-retry (content-addressed media); reuses `promote_path_to_raw` as-is. Phase-A's `Inbox-WIKI/raw/` sidecar is left in place as the routing audit trail — Phase-B writes a *fresh* `<Domain>-WIKI/raw/<ts>_<source>.<ext>` from the same `user_text`/`media_paths`, no coupling to the Phase-A sidecar path (avoids a race and a `Router` Protocol change).
3. **Where the new logic lives.** → **New pure module `src/ai_steward_wiki/inbox/route.py`** (`RouteTarget`/`RouteRejection`/`StagedRaw`; `resolve_target_wiki`, `stage_raw_into_wiki`, `pick_domain_overlay`, `build_ingest_prompt`) + a runtime `_LibrarianAdapter` behind a narrow `Librarian` Protocol (`ingest(decision, …) -> IngestOutcome`). Mirrors Phase-A (`inbox/router.py` + `_RouterAdapter` + `Router`), keeps the Stage-1a and Stage-1b adapters distinct, makes the resolve/move/prompt-build helpers unit-testable in isolation, and gives Phase-C a clean confirm-gate seam between `router.route()` and `librarian.ingest()` in the pipeline.
4. **User reply on success.** → **`decision.notes + "\n\n" + ingest_summary`**, delivered via `OutputDelivery.deliver` (the hybrid-size policy; `run_id` comes from `run_wiki_session`). The user sees both "куда" (Router notes) and "что записал" (Stage-1b summary) — these are not duplicates. Stage-1b `WikiRunnerError` → `send_message(decision.notes + "Не удалось разложить по полочкам — попробую позже.")` (raw kept). Cap/bad-name → `send_message(decision.notes + <ru hint>)`, no ingest. `CLARIFY`/`REJECT` → unchanged Phase-A notes-echo.
5. **`template_id` for `create_wiki`.** → **Always `"_default"`** in Phase-B. The Router emits only a name; a domain→template mapping (and full template-body rendering — a pre-existing gap in `create_wiki`) is a later refinement, ideally driven by extending the Router or a separate domain classifier.
6. **Stage-1b timeout.** → **Always `timeout_s=None`** (the configured text timeout, ~300s), regardless of source. Stage-1b is a full WIKI turn (read input + atomic page edits + `log.md` append), not a short vision-classify; the D-022 vision timeout (~30s) would falsely kill legitimate ingests. No new timeout knob.

## Consequences

1. New module `M-INBOX-ROUTE` in the knowledge graph; `M-TG-PIPELINE-CLASSIFIER` and `M-RUNTIME-WIRING` gain the dependency; new log anchors `tg.pipeline.route.ingest_dispatched|delivered` and `inbox.route.target_resolved|route_target_was_missing|cap_reached|bad_name|raw_moved|ingest.begin|ingest.done|ingest_failed`. Phase-A's `tg.pipeline.router.delivered` was renamed to `tg.pipeline.router.decided` to disambiguate from the new `tg.pipeline.route.delivered`.
2. A routable message now triggers **two** Claude CLI invocations (Stage-1a router + Stage-1b librarian) — latency and token cost roughly double for routable turns. Accepted (the spec's two-stage design, §8.3.3); Phase-E's `## Inbox hint` fast-path skips Stage-1a on confident hints; Phase-B adds no third run.
3. **Phase-B auto-executes route+ingest with no user confirmation** — the inline-button confirm loop is Phase-C (`aisw-e45`), which wraps the resolve+move+ingest behind a gate at the `router.route() → librarian.ingest()` seam in the pipeline. Acceptable for an MVP slice: the WIKI is the user's own private sandbox; Stage-1b only does atomic page writes + `log.md` append (per `prompts/wiki.md`); per-WIKI git history (when `M-OPS-BACKUP` is wired) gives undo.
4. **With Phase-A + Phase-B both done, the "do not deploy `master`" caveat from ADR-003 is lifted** — the route+ingest loop is closed. (Recorded on epic `aisw-t2r`.)
5. Phase-D builds on the same orchestration point: a `RouterDecision` carrying a reminder/aggregator spec → `jobs.db` insert via the scheduler, alongside (or instead of) the Stage-1b ingest.
6. Open follow-ups (not Phase-B): domain→template_id mapping + full template-body rendering in `create_wiki`; per-WIKI git auto-commit on ingest; retry/sweep of un-ingested files left in `<Domain>-WIKI/raw/`; restore-from-trash on a `ROUTE` to a soft-deleted name.
