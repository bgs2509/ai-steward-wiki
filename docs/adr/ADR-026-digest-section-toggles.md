# ADR-026: Inbox-WIKI Phase-D.b.2c — per-user digest section toggles: `user_digest_prefs` (sessions.db, two-boolean row, FK `users.user_id` CASCADE); `alembic/sessions/0002` as an idempotent `Base.metadata.create_all` delta (incremental-migration convention); `/digest_sections` inline-keyboard toggle command + `digestsec:` callback; `fire_digest_job` injects a «Не включай разделы: …» directive only when a section is off (byte-identical default); `/expand` ignores the toggle

**Status:** accepted
**Date:** 2026-05-13
**Context:** [D-024](../Spec-WIKI/decisions/D-024-digest-format.md), [D-031](../Spec-WIKI/decisions/D-031-identity.md), [D-042](../Spec-WIKI/decisions/D-042-identity-vocabulary.md), [D-032](../Spec-WIKI/decisions/D-032-ru-only-mvp.md); [ADR-007](ADR-007-inbox-wiki-digest-job.md), [ADR-024](ADR-024-digest-presentation.md), [ADR-025](ADR-025-digest-interactive-surface.md); bd `aisw-pv8` (Phase-D.b.2c), parent `aisw-19o` (Phase-D.b umbrella), sibling `aisw-269` (Phase-D.b.2b — digest interactive surface, closed), epic `aisw-t2r`. Discovery/design/plan: `docs/superpowers/specs|plans/20260512-inbox-wiki-digest-section-toggles-{discovery,design,plan}.md`.

## Context

`aisw-269` (Phase-D.b.2b, [ADR-025](ADR-025-digest-interactive-surface.md)) shipped the digest *interactive surface* — `/digest_now`, `/expand <section>`, `DigestPayload.wiki_scope` named-subset — and at its Q&A gate **deferred two candidate features** to their own bds (ADR-025 §8):

- **Actionable inline cards** (medication-due-now / event-soon / pending_confirmation) — no real data source exists in the current architecture (`reminder_job` has no medication/event subtype or done-state; `sessions.PendingConfirm` has a 10-minute TTL so a morning/evening cron digest essentially never finds one still `pending`). Still deferred; revisited when there is a job-category model or a long-lived "needs your answer" queue.
- **Per-user section toggles** (`trackers` on/off, `wiki` on/off) — *"a toggle table with no flip surface is inert, and the flip UX **plus the first `sessions.db` Alembic migration past `0001_baseline`** want their own design pass."* — **this ADR / `aisw-pv8`** is that design pass.

The increment is small (one new `sessions.db` table, the first incremental `sessions.db` migration, the command that flips it, the read that honours it in the digest run) but it sets two precedents — the incremental-migration style for `sessions.db`, and the bot's third slash command — so the shape decisions are worth recording.

## Options & decisions

1. **Storage shape — generic per-section rows, or a fixed two-boolean row?** → **One row per owner with two boolean columns**: `user_digest_prefs(user_id INTEGER PK FK→users.user_id ON DELETE CASCADE, trackers_enabled BOOLEAN NOT NULL DEFAULT 1, wiki_enabled BOOLEAN NOT NULL DEFAULT 1, updated_at_utc DATETIME NOT NULL)`. The toggleable set is closed by ADR-025 (exactly `trackers` and `wiki`; TL;DR and `today` stay always-on — TL;DR *is* the summary, `today` is the planner-window core); a generic `(user_id, section_key, enabled)` EAV table would carry JOIN/upsert plumbing, lose schema-level defaults and type safety, and need a key CHECK anyway — over-engineering for two flags (YAGNI). Absent row (or absent `User` row) ⇒ both `True` — the feature is opt-out, never opt-in, so a user who never touches `/digest_sections` sees zero change. FK target is `users.user_id` (the surrogate), mirroring `inbox_hint_cache`, resolved from `telegram_id` via the existing `storage.sessions.users.resolve_user_id`; `ON DELETE CASCADE` means a deleted `User` takes its prefs with it and the next read returns the safe default. The repo lives in a new `storage/sessions/digest_prefs.py` (`DigestPrefs` frozen dataclass with a `disabled_keys` property; `get_digest_prefs` read-only; `set_digest_section` upsert — creates the row with defaults if absent, bumps `updated_at_utc`, no-op + defaults if the `telegram_id` has no `User` row so the command never crashes if invoked before onboarding, `ValueError` on an unknown section key). **Rejected:** a generic EAV table; a JSON `sections_json` column (loses schema and indexing for no gain).

2. **`alembic/sessions/0002` — bare `op.create_table`, or an idempotent `create_all`?** → **`upgrade()` = `Base.metadata.create_all(bind=op.get_bind())`** (`checkfirst=True` by default), **`downgrade()` = `op.drop_table("user_digest_prefs")`**, `down_revision = "0001_sessions_baseline"`. `0001`'s `upgrade()` is `Base.metadata.create_all` of **live** metadata — so once `UserDigestPrefs` is added to `models.py`, a from-empty `alembic upgrade head` already creates `user_digest_prefs` in the baseline step; a bare `op.create_table("user_digest_prefs")` in `0002` would then raise *"table already exists"* on every fresh DB. An idempotent `create_all` delta does work only on a DB that was baselined before the model existed, no-ops otherwise, and the stepwise (`0001`→`0002`) and from-empty paths converge. **CONVENTION recorded here for all `sessions.db`/`jobs.db`/`audit.db` migrations:** a migration that only *adds* tables → add the ORM model to `models.py` (so the baseline's `create_all` keeps producing the whole current schema) **and** a new `NNNN_*` whose `upgrade()` calls `Base.metadata.create_all(bind=op.get_bind())` and whose `downgrade()` explicitly drops the added table(s); a migration that *ALTERs or drops existing columns* → explicit `op.*` batch ops (`render_as_batch=True` is already set in each `env.py`). **Rejected:** rewriting `0001_baseline` (and, for consistency, the `jobs`/`audit` baselines) to explicit `op.create_table` per table — touches working baselines, risks divergence from what `create_all` actually produces, and is scope creep ("don't refactor what you're not changing"); the `create_all`-snapshot + `op.*`-delta pattern is kept and merely made explicit.

3. **`/digest_sections` UX — inline-keyboard toggles, or text arguments?** → **An inline keyboard of toggle buttons.** `/digest_sections` replies with a ru header and one button per `TOGGLEABLE_DIGEST_SECTIONS` key — text `«📈 Трекеры: вкл ✅»` / `«📈 Трекеры: выкл ⬜»`, `callback_data` `digestsec:<section>:<0|1>` carrying the **target** state (so a tap on a stale message is idempotent). A new `@router.callback_query(F.data.startswith("digestsec:"))` handler: `parse_digestsec_callback` (split into exactly 3 on `:`, `section ∈ TOGGLEABLE_DIGEST_SECTIONS`, flag ∈ `{"0","1"}`; else `None`) → `firing.set_owner_digest_section` → `cb.message.edit_reply_markup` with the rebuilt keyboard (the edit is wrapped — a `TelegramBadRequest "not modified"` on a stale re-tap is swallowed, `tg.command.digest_sections.edit_skipped`, since the DB write already succeeded) → `cb.answer`. The new `digestsec:` prefix is independent of the existing `confirm:` prefix/parser. `build_router` reaches the prefs through `firing.get_owner_digest_prefs` / `firing.set_owner_digest_section` accessors (backed by the sessions sessionmaker in the digest context — same registry pattern as `firing.list_owner_digest_job_ids` / `run_section_expand`), so `handlers.py` stays thin; it imports `DigestPrefs` / `TOGGLEABLE_DIGEST_SECTIONS` / `SECTION_DISPLAY_NAME` directly from `storage.sessions.digest_prefs` only for rendering the keyboard. Registered in the same commands group as `/digest_now` and `/expand` (before the `F.text` catch-all; the router-order regression test still holds). Inline-keyboard "settings panel" is the standard Telegram-bot UX for togglable settings (no exact keys to remember, immediate visual state); text-args is the dated fallback. Ru-only strings, no i18n catalog (D-032). **Rejected:** `/digest_sections <key> <on|off>` text args (clunky on mobile, must remember the keys, no visual state); a hybrid command accepting both (double the input surface for marginal gain, YAGNI).

4. **Honouring the toggles in the digest — append a directive line, or a dedicated prompt template variable, or a new runner argument?** → **Append a `«Не включай разделы: …»` line to the existing `planner_context`/`user_input` string, only when the disabled set is non-empty.** In `fire_digest_job`, after the planner context is built and before the runner call: read `user_digest_prefs` (wrapped — any failure → `scheduler.digest.prefs_read_failed` + `DigestPrefs(True, True)`, i.e. degrade-to-all-on, never skip the digest over a prefs read; `fire_digest_job` keeps its outer catch-all so the scheduler loop survives regardless); if `prefs.disabled_keys` is non-empty → `planner_context += f"\n\nНе включай разделы: {', '.join(SECTION_DISPLAY_NAME[k] for k in disabled)}."` and log `scheduler.digest.sections_filtered{job_id, owner_telegram_id, disabled}`; otherwise leave `planner_context` untouched — **byte-identical to today** for the overwhelming common case (regression-guarded by a test that the no-prefs path calls the runner with the same string). `prompts/digest.md` (semver `0.1.0 → 0.1.1`) gains one sentence: *«Если в сообщении есть строка вида „Не включай разделы: …" — полностью пропусти перечисленные секции, даже если по ним есть содержимое.»* The `DigestRunner` Protocol and `_DigestRunnerAdapter` are **not** touched (the directive rides inside the existing string; `runner(..., section=None)` is exactly as today). **Rejected:** a dedicated `{disabled_sections}` template variable rendered unconditionally (would change the prompt input even when nothing is off — perturbs the digest golden tests and loses byte-identity; to keep identity you'd end up "render only when non-empty" anyway, but with a heavier render path); a new `disabled: list[str]` argument on the `DigestRunner` Protocol (just-widened with `section=` in `aisw-269`; the adapter would have to turn it into text for Claude anyway — more moving parts for the same result).

5. **`/expand <section>` vs the toggle — does an explicit `/expand trackers` work when `trackers` is off?** → **Yes — an explicit `/expand` overrides the toggle; `run_section_expand` / the `/expand` path is unchanged.** The toggle governs only the unattended cron digest; "show me X right now, I'm asking" must not be blocked by "don't put X in the automatic rollup" (same UX principle as muting a channel not blocking you from reading it). This also keeps the `/expand` code and its tests untouched and avoids reading `user_digest_prefs` in a third place. **Rejected:** `/expand trackers` returning a ru hint *«раздел отключён в /digest_sections»* when `trackers` is off — consistent in a narrow sense, but blocks a one-off peek and forces an enable→view→disable dance.

## Consequences

1. **`M-STORAGE-SESSIONS`** (`storage/sessions/models.py` + new `storage/sessions/digest_prefs.py`): NEW `UserDigestPrefs` ORM model (two-boolean row, FK `users.user_id` CASCADE); NEW `DigestPrefs` frozen dataclass (+`disabled_keys`), `TOGGLEABLE_DIGEST_SECTIONS = ("trackers", "wiki")` (a subset of `M-TG-HANDLERS-WIRING.EXPAND_SECTION_KEYS` — a test asserts the subset relation), `SECTION_DISPLAY_NAME` (key → ru label with emoji), `get_digest_prefs` (read), `set_digest_section` (upsert; no-op + defaults if no `users` row; `ValueError` on unknown section). New `MODULE_CONTRACT`/`MODULE_MAP` entries; `models.py` `VERSION` bumped.
2. **`alembic/sessions/versions/0002_user_digest_prefs.py`**: NEW — `upgrade()` = idempotent `Base.metadata.create_all`, `downgrade()` = `op.drop_table("user_digest_prefs")`, `down_revision = "0001_sessions_baseline"`. `tests/unit/storage/test_baselines.py`: `user_digest_prefs` added to the `sessions` expected-tables set + a stepwise-upgrade test (stamp `0001` then `upgrade head`). No `jobs.db` / `audit.db` change.
3. **`M-SCHEDULER-FIRING`** (`scheduler/firing.py`): `set_digest_context` gains `sessions_session_maker` (the digest-context tuple goes 6 → 7); NEW `get_owner_digest_prefs` / `set_owner_digest_section` accessors (read the sessions sessionmaker from the context; `DigestNotInitialisedError` if unwired); `fire_digest_job` reads the owner's prefs (degrade-to-all-on) and, when a section is off, appends the `«Не включай разделы: …»` directive + logs `scheduler.digest.sections_filtered` — byte-identical otherwise; new anchors `scheduler.digest.sections_filtered` / `scheduler.digest.prefs_read_failed`. The `DigestRunner` Protocol, `_DigestRunnerAdapter`, `run_section_expand` (the `/expand` path) are untouched. New `DEPENDS` on `M-STORAGE-SESSIONS`.
4. **`M-TG-HANDLERS-WIRING`** (`tg/handlers.py`): NEW `Command("digest_sections")` handler + `@router.callback_query(F.data.startswith("digestsec:"))` handler (both wrapped — nothing bubbles to the dispatcher); NEW `parse_digestsec_callback`, `_build_digest_sections_kb`, `DIGESTSEC_CALLBACK_PREFIX = "digestsec:"`, ru strings; registered in the same commands group as `/digest_now`, `/expand`. New anchors `tg.command.digest_sections.{shown,toggled,bad_callback,edit_skipped,unavailable,error}`. New `DEPENDS` on `M-STORAGE-SESSIONS`.
5. **`M-RUNTIME-WIRING`** (`__main__.py`): `firing.set_digest_context(...)` now also passes the sessions sessionmaker.
6. **`prompts/digest.md`**: `semver 0.1.0 → 0.1.1`; one sentence describing the `«Не включай разделы: …»` directive. `prompts/digest_expand.md` untouched.
7. **GRACE**: `knowledge-graph.xml` (`M-STORAGE-SESSIONS` / `M-SCHEDULER-FIRING` / `M-TG-HANDLERS-WIRING` annotations + `DEPENDS` + new CrossLinks `M-TG-HANDLERS-WIRING → M-STORAGE-SESSIONS`, `M-SCHEDULER-FIRING → M-STORAGE-SESSIONS`); `verification-plan.xml` (`V-M-STORAGE-SESSIONS` / `V-M-TG-HANDLERS-WIRING` / `V-M-SCHEDULER-FIRING` tests + markers + evidence); `development-plan.xml` (new `Phase-D.b.2c` entry, `Phase-D.b.2b` → `done`).
8. **No new third-party dependency. Exactly one new SQLite table (`user_digest_prefs`) and one new Alembic migration (`alembic/sessions/0002`), both `sessions.db` only.** `DigestPayload` / `storage/jobs/*` untouched (no `jobs.db` migration); `tg/pipeline.py` untouched.
9. **Out of scope (→ own future bds / carried over):** actionable inline cards (still deferred per ADR-025 §8 — needs a job-category model or a long-lived "needs your answer" queue first); scheduling-time section selection (picking sections when creating the digest job, the way `wiki_scope` is picked — `DigestPayload` / `jobs.db` left untouched on purpose); per-WIKI-per-section toggles; toggles for the always-on sections (TL;DR, `today`); a `meds` toggle (the medication section is auto-omitted when empty, so the value-add is marginal — revisit only if asked); rewriting `0001_baseline` (and the `jobs`/`audit` baselines) to explicit `op.create_table`; digest management UX (`/jobs_list`, cancel/snooze/edit of a digest job); the `asyncio.PriorityQueue` worker-loop consumer; the `jobs.jobs ↔ APScheduler` reconciler; `data/runs/` retention; i18n.
10. **Rejected alternatives:** a generic EAV `user_digest_prefs` table; a bare `op.create_table` in `0002` (conflicts with the baseline's `create_all` on fresh DBs); rewriting the baselines to explicit table creation; `/digest_sections <key> <on|off>` text args (or a hybrid); a `{disabled_sections}` prompt template variable / a new `DigestRunner` argument for the directive; `/expand` respecting the toggle; extending ADR-025 instead of a new ADR (the storage shape + the incremental-migration convention + the inline-toggle UX + the prompt-injection mechanism + the `/expand`-override are a distinct decision cluster from ADR-025's interactive-surface decisions).

## Перенос в ADR (D-024)

This ADR closes the "per-user section toggles" half of the «перенос в ADR» deferral that [ADR-025](ADR-025-digest-interactive-surface.md) §8 carried out of [D-024](../Spec-WIKI/decisions/D-024-digest-format.md) — it records the `user_digest_prefs` data model, the first incremental `sessions.db` migration and its convention, the `/digest_sections` flip UX, the digest-prompt directive mechanism, and the `/expand`-override decision. The D-024 actionable-cards table remains deferred (ADR-025 §8).
