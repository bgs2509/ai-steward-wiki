# ADR-025: Inbox-WIKI Phase-D.b.2b ŌĆö digest interactive surface: first slash-command router (`/digest_now`, `/expand <section>`); `DigestRunner(section=ŌĆ”)` scoped re-run; `DigestPayload.wiki_scope` widened to `'all' | list[str]`; cards + per-user toggles deferred

**Status:** accepted
**Date:** 2026-05-12
**Context:** [D-024](../Spec-WIKI/decisions/D-024-digest-format.md), [D-023](../Spec-WIKI/decisions/D-023-confirmation-flow.md), [D-032](../Spec-WIKI/decisions/D-032-ru-only-mvp.md), tech-spec ┬¦3/┬¦6; [ADR-006](ADR-006-inbox-wiki-reminder-cron-bridge.md), [ADR-007](ADR-007-inbox-wiki-digest-job.md), [ADR-024](ADR-024-digest-presentation.md); bd `aisw-269` (Phase-D.b.2b), sibling `aisw-w3k` (Phase-D.b.2a ŌĆö presentation core, closed), parent `aisw-19o` (Phase-D.b umbrella), epic `aisw-t2r`. ## Context

`aisw-w3k` (Phase-D.b.2a, ADR-024) shipped the digest *presentation core* ŌĆö `fire_digest_job` ŌåÆ `deliver_output(kind='digest')`, the D-024 prompt contract, the real `jobs.db` planner-window query ŌĆö and explicitly deferred the *interactive* half (cards, `/expand`, `/digest_now`, per-user toggles, named-subset WIKI) to **aisw-269**, the second half of the Plan-Sizing split of `aisw-19o`.

At the aisw-269 Q&A gate (2026-05-12) the interactive half was **re-scoped**: of the five candidate features, two were deferred and three kept.

- **Actionable inline cards (medication-due-now / event-soon / pending_confirmation)** were dropped because no real data source exists in the current architecture: `reminder_job` (ADR-006) is one free-text reminder with `lead_time_min` and **no medication/event subtype and no done-state**, so its "cards" would have nothing to be sourced from or resolved against; `sessions.PendingConfirm` (D-023) *is* stateful and button-resolvable but has a **10-minute TTL**, so a morning/evening cron digest essentially never finds one still `pending`. A card surface here would be near-dead code. Cards are revisited when there is either a job-category model or a long-lived "needs your answer" queue.
- **Per-user section toggles** (`tracker on/off`, `wiki-updates on/off`) were dropped because a toggle table with no flip surface is inert, and the flip UX **plus the first `sessions.db` Alembic migration past `0001_baseline`** want their own design pass; shipping `user_digest_prefs` now would be a half-finished implementation.

That leaves `/digest_now`, `/expand <section>`, and the contained `DigestPayload.wiki_scope` widening ŌĆö the two highest-value, lowest-risk owner controls plus the schema change that lets a recurring digest be scoped at creation time.

## Options & decisions

1. **Slash commands ŌĆö how to add the bot's first command surface?** The bot has zero `Command`-filter handlers today; everything is NL ŌåÆ `MessagePipeline`. ŌåÆ **A dedicated `aiogram.filters.Command` group registered BEFORE the existing `F.text` catch-all** (in `tg/handlers.py`'s `build_router`, or a small `tg/commands.py` it includes). aiogram matches handlers in registration order, so commands must come first; an ordinary text message still falls through to `MessagePipeline.on_text` unchanged. A router-order regression test guards this. No new dependency (`aiogram.filters.Command` is part of aiogram 3.x). Rejected: a `/`-prefix check inside the existing text handler (fragile, mixes concerns).

2. **`/digest_now` semantics ŌĆö run the configured digest job(s), or an ephemeral all-WIKIs digest?** ŌåÆ **Run all the calling owner's enabled `digest_job` rows now, via the existing `fire_digest_job(job_id)` per row.** This reuses the entire Phase-D.b.1/2a pipeline as-is: `set_digest_context`, the per-WIKI `LockAcquirer` held inside `run_wiki_session`, `deliver_output(kind='digest', job_id=ŌĆ”)`, `_build_planner_context`, the 3-strike auto-disable, the `wiki_scope`/`window_hours` the owner already chose. A concurrent cron digest for the same WIKI just queues behind the per-WIKI flock ŌĆö accepted (no new lock). Zero digest jobs ŌåÆ a ru hint to create one. A per-job exception is caught so the remaining jobs still run. **Rejected:** refactoring `fire_digest_job` into a reusable `_run_digest` core + an ephemeral path (broadens the diff and test surface; `deliver_output` requires a `job_id` for its audit row, so an ephemeral run would force either a nullable `job_id` schema change or a transient `Job` row ŌĆö both heavier than the chosen reuse).

3. **`/expand <section>` ŌĆö re-run Claude scoped to a section, or replay the persisted digest file?** ŌåÆ **Re-run Claude scoped to one section** via a generalised `DigestRunner` Protocol that gains `section: str | None = None` ŌĆö `None` ŌćÆ the full D-024 digest (`prompts/digest.md`, today's behaviour, byte-identical), a key ŌćÆ section detail from a new `prompts/digest_expand.md` (the section name substituted into `user_input`). Section keys mirror the four D-024 `<b>`-headers: `today` / `meds` / `trackers` / `wiki`; unknown/missing ŌåÆ a ru usage line. The reply goes via `sender.send_message` ŌĆö an on-demand command reply is not a cron output, so it gets **no `run_outputs` audit row** (unlike the digest itself). "Nothing for this section over the period" is a normal reply, not an error. `_DigestRunnerAdapter` in `__main__` swaps `overlay_prompt_path` and `user_input` on `section`; `planner_context` is unused in expand mode. **Rejected:** slicing the `<b>`-section out of the persisted `<wiki>/data/runs/<date>/<run_id>.md` ŌĆö that returns exactly what was already in the digest, which is not "expand", and depends on a fresh file existing.

4. **`DigestPayload.wiki_scope` widening.** ŌåÆ `Literal['all']` ŌåÆ `'all' | list[str]` (the list constrained non-empty), `VERSION 0.0.4 ŌåÆ 0.0.5`. **No `jobs.db` Alembic migration** ŌĆö `'all'` stays a valid value so existing rows keep validating, and `wiki_scope` is not the union discriminator (`kind` is), so the discriminated-union schema is unaffected. `create_digest_job` accepts `wiki_scope: str | list[str]` (its `"all" if ŌĆ” else ŌĆ”` branch is already there). The digest fast-path in `tg/pipeline.py` extracts the name list **heuristically** ŌĆö whole-token, case-insensitive intersection of the NL turn with the owner's `*-WIKI/` dir-stems; all mentioned names resolve ŌćÆ `list[str]`, none mentioned ŌćÆ `'all'`, a name-shaped token that does not resolve ŌćÆ a ru clarification and **no job created** (fail-fast at the boundary ŌĆö never persist a `wiki_scope` that points at nothing). `fire_digest_job`, after resolving the owner's WIKI set, intersects it with `payload.wiki_scope` when that is a list (logs `scheduler.digest.scope_filter` with `requested`/`kept`/`vanished`); an empty intersection ŌćÆ a ru "all scoped WIKIs vanished" notice, `finished_at_utc` set, status stays `'scheduled'`, **no strike** (config drift, not a run failure). The recap/ack names the chosen WIKIs when scoped. **Rejected:** Haiku-assisted name extraction (the candidates are the owner's own dir-stems ŌĆö a finite known set; a Haiku call on the job-creation path is overkill, and the discovery for `aisw-oqq` already said "heuristic enough").

## Consequences

1. **`M-TG-HANDLERS-WIRING`** (`tg/handlers.py`, or new `tg/commands.py`): NEW `Command('digest_now')` / `Command('expand')` handlers registered before `F.text`; both wrap their body in `try/except` ŌåÆ a ru error reply, never bubble to the dispatcher; `/digest_now` catches per-job exceptions. New log anchors `tg.command.digest_now`(`.empty`/`.job_failed`/`.done`), `tg.command.expand`(`.bad_section`/`.delivered`/`.failed`).
2. **`M-SCHEDULER-FIRING`** (`scheduler/firing.py`): `DigestRunner` Protocol `+section: str | None = None` (`fire_digest_job` calls it with the default ŌĆö byte-identical); `fire_digest_job` intersect-and-filter on `payload.wiki_scope` (new anchor `scheduler.digest.scope_filter`; empty kept ŌćÆ ru notice, `scheduler.digest.delivered` `empty='scope_vanished'`, no strike); `create_digest_job` `wiki_scope: str | list[str]`; `scheduler.digest.scheduled` carries `wiki_scope`.
3. **`M-STORAGE-JOBS`** (`storage/jobs/payloads.py`): `DigestPayload.wiki_scope: Literal['all'] | list[str]` (non-empty), `VERSION 0.0.4 ŌåÆ 0.0.5`. **No `jobs.db` migration.**
4. **`M-TG-PIPELINE-CLASSIFIER`** (`tg/pipeline.py`): digest fast-path gains `_extract_wiki_names`; the `category='digest'` confirm draft carries the new `wiki_scope` shape; the recap/ack names the WIKIs when scoped; new anchor `tg.pipeline.digest.wiki_unknown`.
5. **`M-RUNTIME-WIRING`** (`__main__.py`): `_DigestRunnerAdapter.run(section=None)` ŌĆö swap `overlay_prompt_path` (to `prompts/digest_expand.md`) and `user_input` (to the section name) when `section` is set; wire `digest_expand_prompt_path = settings.prompts_dir / "digest_expand.md"` into the adapter ctor; `/digest_now` and `/expand` handlers wired with the firing context / `resolve_owner_wikis` / the digest runner.
6. **New artifact** `prompts/digest_expand.md` (`semver 0.1.0`, ru, HTML whitelist as `digest.md`, "nothing for this section over the period" one-liner) ŌĆö a knowledge-graph node referenced by `M-RUNTIME-WIRING`.
7. **No new third-party dependency; no new SQLite table; no new Alembic migration.**
8. **Out of scope (ŌåÆ own future bds):** actionable inline cards (need a job-category model or a long-lived "needs your answer" queue first); per-user section toggles + `user_digest_prefs` + `alembic/sessions/0002_*` (need the flip UX + the first `sessions.db` migration past baseline). Out of scope project-wide / carried over: `_run_digest` core refactor; an ephemeral `/digest_now` without a configured job; Haiku-assisted name extraction; a `run_outputs` row for `/expand`; digest management UX (`/jobs_list`, cancel/snooze/edit); the queue consumer; the `jobs.jobs Ōåö APScheduler` reconciler; `data/runs/` retention; i18n.
9. **Rejected alternatives:** a `/`-prefix check inside the text handler; a `_run_digest` core refactor + ephemeral `/digest_now`; replaying the persisted digest file for `/expand`; Haiku-assisted WIKI-name extraction; extending ADR-024 instead of a new ADR (the slash-command router + scoped re-run + schema widening + the two deferrals are a distinct decision cluster from ADR-024's presentation-format decisions).
