### Open questions

- `src/ai_steward_wiki/scheduler/consumer.py:222` — `asyncio.wait_for(proc.communicate(), timeout=self._timeout_s)`. When timeout fires, `wait_for` raises `asyncio.TimeoutError` and cancels `proc.communicate()`. Does the subprocess itself get SIGKILL? The code calls `systemd-run --scope --collect`, but timeout raises before any explicit `proc.terminate()/kill()` call is made. If the asyncio task is cancelled but systemd scope is already running, the scope may outlive the timeout. The existing SIGKILL comment in the spec (NFR-2: "Consumer kills scope") may rely on systemd handling this — but it's worth explicitly verifying that `--collect` cleans up on consumer-side cancellation.
- `src/ai_steward_wiki/wiki/lifecycle.py:122` — `wiki_root` is set from `Settings.wiki_root` via injection. The wiki_id comes from classifier output. There is no `is_relative_to` check verifying the resolved path stays under `wiki_root`. Given that `wiki_id` goes through `normalize_wiki_name` (strict `^[A-Z][A-Za-z0-9]*-WIKI$` regex), this is unlikely to be exploitable. But confirming the contract explicitly (resolved path must be under `wiki_root`) would be worth a unit test.
