### Open questions

1. **Concurrent cron fire — double-enqueue — `src/ai_steward_wiki/scheduler/cron_user.py:156-194`.** The fire transaction reads `row.status=='scheduled'`, calls `queue.put(...)`, then writes `status='queued'` in one `session.begin()`. No test exercises two simultaneous `CronTrigger` fires for the same job. Under overlapping triggers both could pass the status guard before either commits → the job is enqueued twice. The consumer's own idempotency guard likely makes this harmless, but the at-most-once claim (NFR-1) isn't proven by a test. Confirm intent or add a concurrent-fire test.

2. **User command reaches CLI argv unfiltered — `src/ai_steward_wiki/scheduler/consumer.py:301`.** `msg.command` (raw text after `|` in `/cron_add`) is appended to the `systemd-run … -- claude … <command>` argv. This is **not** a shell-injection bug — `asyncio.create_subprocess_exec` with a list argv and no `shell=True` passes it as one literal token (verified across consumer.py and wiki/runner.py; no `shell=True` anywhere). The open question is product-side: there is no length cap or content policy on `command`, and it is executed by an authenticated Claude CLI scope. Confirm a max-length / rate guard is intended (discovery R-6 defers per-user rate limit to a follow-up).
