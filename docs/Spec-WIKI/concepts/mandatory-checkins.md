# Mandatory Check-ins

**Тип:** concept
**Статус:** draft
**Источники:** диалог 2026-05-08, [time-tracker](../entities/time-tracker.md)

## Суть

Для **обязательных дел** (зарядка, еда, сон, учёба, таблетки, лекарства, диета) Клод не просто напоминает — он **проверяет факт выполнения** через follow-up через N минут.

## Жизненный цикл одного check-in

1. **T₀ — напоминание.** «Сейчас зарядка» / «Прими таблетку X».
2. **T₀ + N мин — follow-up.** «Сделал зарядку?» с двумя кнопками: ✅ Да / ❌ Нет.
3. **Ответ:**
   1. ✅ — фиксируется как `done`, идёт в историю.
   2. ❌ — спросить «отложить на 15 мин или пропустить?». При «отложить» — повторить с T₀ = now+15. При «пропустить» — `skipped`, в историю.
   3. Нет ответа за M минут — `missed`, в историю, на следующий день поднять приоритет.

## Категории

1. **Зарядка** — обычно утром, follow-up через 30 мин.
2. **Еда** — завтрак/обед/ужин, follow-up через 45 мин.
3. **Сон** — отбой и подъём, follow-up через 30 мин (фактическое время).
4. **Учёба** — по расписанию из `planner.json`, follow-up через 60 мин.
5. **Таблетки / лекарства** — критично, follow-up через 15 мин, при `missed` — повторное напоминание.
6. **Диета** — фиксация по приёмам пищи, без follow-up как такового (логирование).

N (delay до follow-up) — поле категории, не глобальная константа.

## Anti-spam: подавление общего опроса

Если в окне **±30 минут** от слота periodic-опроса (см. [schedule-profiles](schedule-profiles.md)) уже есть mandatory-checkin или planner-item — общий опрос «что делал?» **не отправляется**. Юзер уже что-то скажет.
