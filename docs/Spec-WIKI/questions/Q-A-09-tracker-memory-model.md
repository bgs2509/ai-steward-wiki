# Q-A-09: Где хранить паттерны юзера для predictive-replies?

**Tier:** A (архитектурный, требует решения до реализации)
**Источник:** диалог 2026-05-08, [predictive-replies](../concepts/predictive-replies.md)

## Формулировка

Predictive-replies нужно знать «что юзер обычно делал по понедельникам в 14:00». Это требует **хранилища истории ответов** + агрегатора паттернов. Где это живёт в архитектуре `ai-steward-wiki`?

## Варианты ответа

1. **A — Отдельный файл `patterns/weekly.json` в WIKI юзера**
   1. Структура: `{ "monday": { "14:00": ["работа", "обед", "дорога"] }, ... }`
   2. ✅ Простой формат, читаемый человеком.
   3. ✅ Изолирован от `jobs.db` — не трогает job-model.
   4. ❌ Нужен отдельный агрегатор (cron-job: раз в сутки пересчитать топ-3 по слотам).
   5. ❌ Лишний файл рядом с WIKI-содержимым; runtime всё равно читает его как state.
2. **B — Обогащение `jobs.payload` полем `answer_history`**
   1. Каждый tracker/job item получает `answer_history: [...]`.
   2. ✅ Один storage-контур — `jobs.db`.
   3. ❌ Смешивает «план» и «факт» в одном row — нарушает SoC.
   4. ❌ Раздувает payload и усложняет retention.
3. **C — Агрегат в `_output/tracker_patterns.md`**
   1. Markdown-отчёт с таблицей паттернов, обновляется ежедневно.
   2. ✅ Соответствует конвенции `_output/` (генерируемые отчёты).
   3. ❌ Markdown — не лучший формат для чтения программой при каждом опросе.
   4. ❌ Парсинг таблицы хрупкий.
4. **D — База ответов + on-the-fly агрегация**
   1. Append-only лог `data/tracker_log.jsonl`: `{ts, slot, answer}`.
   2. Топ-3 считается на лету при отправке вопроса (запрос за последние N недель, group by slot).
   3. ✅ Минимум структур — лог + чистая функция.
   4. ✅ Легко эволюционировать (добавить recency-bias, исключения).
   5. ❌ Чуть дороже на каждый опрос (читать N недель лога).
   6. ❌ Требует определиться с retention.

## Решение

**Принято 2026-05-08:** append-only таблица `tracker_answers` в jobs.db (вариант D из Q переформулирован как A в финальной редакции). Топ-3 — индексированный SQL on-the-fly, retention 90 дней, slot-granularity (dow, hour), recency-bias отложен в v2.

Параметры:
1. Юзер не редактирует паттерны руками — это derived data из лога ответов.
2. Retention: 90 дней rolling.
3. Агрегатор: SQL в самом боте при отправке вопроса; pruning — daily maintenance-job.
