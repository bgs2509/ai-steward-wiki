# Q-B-11: Идемпотентность ingest

**Tier:** B
**Источник:** [overview §9 п.11](../raw/20260507-ai-steward-wiki-only-overview.md)

## Формулировка

Как избежать дубля напоминаний если юзер кинул один и тот же билет дважды? Hash-чек контента / dedup-окно / явное «уже видел».

## Варианты

1. **A. SHA-256 hash контента.** Таблица `seen_files(hash, wiki, first_seen)`. Простой и детерминированный.
2. **B. Dedup-окно по времени.** Пропуск, если такой же тип контента приходил за последние N минут. Хрупко.
3. **C. LLM-сравнение.** Router смотрит — «ты уже кидал похожее, добавить или skip?». Дорого.
4. **D. Гибрид A+C.** Hash как fast-check; при коллизии метаданных — спросить юзера.

## Решение

- [x] **Вариант E** — двухслойный dedup без LLM:
  - **L1 (TG idempotency):** `(chat_id, update_id)` в `audit.db`, TTL 24h — защита от webhook-retry, прозрачно.
  - **L2 (content hash):** SHA-256 нормализованного контента (текст: trim+lower; файл: bytes) в `jobs.db.seen_files(hash PK, wiki, first_seen, tg_message_id)`, TTL 30d. На совпадении — inline-кнопка «уже видел такое N дней назад в `<WIKI>`. Создать ещё раз / открыть существующий?».
  - **L3 (LLM-сравнение)** не делаем в MVP; добавим позже точечно на этапе job-creation, если семантические дубли начнут мешать.
- [x] оформлено как [D-018](../decisions/D-018-ingest-idempotency.md)
